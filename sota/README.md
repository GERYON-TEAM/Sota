# SOTA Frontend

Фронтенд-проект на `React + TypeScript + Vite` с набором экранов для нескольких ролей:

- специалист
- заказчик
- валидатор
- авторизация и вспомогательные экраны

Сейчас репозиторий в первую очередь содержит UI, локальную логику экранов и моковые данные. Отдельного слоя API пока нет, поэтому этот README и документы в `docs/` описывают, куда и как безопасно подключать бэкенд дальше.

## Что есть в проекте сейчас

- ручная маршрутизация через `window.location.pathname`
- страницы, собранные по модулям внутри `src/pages`
- локальные хуки для UI-состояния, фильтров, модалок и форм
- моковые данные рядом со страницами и типами
- задел под более строгую модульную архитектуру: `app`, `processes`, `widgets`, `features`, `entities`, `shared`

## Структура директорий

```text
src/
  app/        # пока почти не используется, сюда стоит выносить инициализацию приложения
  entities/   # задел под доменные сущности
  features/   # задел под пользовательские сценарии
  pages/      # основные экранные модули
  processes/  # задел под межстраничные процессы
  shared/     # переиспользуемые UI-компоненты и общие утилиты
  widgets/    # задел под крупные составные блоки
```

Сейчас центр проекта находится в `src/pages`, а слои `entities/features/widgets/processes/app` в основном подготовлены как каркас для следующего этапа архитектурного упорядочивания.

## Куда подключать бэкенд

Короткая рекомендация:

1. Не вставлять `fetch` напрямую в компоненты страниц.
2. Для каждого модуля создать папку `api` рядом со страницей или вынести API в `src/entities`/`src/features`, если логика станет общей.
3. Оставить UI-хуки для локального состояния, а сетевые хуки выделять отдельно.
4. Моки из `types/*.types.ts` постепенно заменять на DTO, мапперы и реальные ответы сервера.

Базовый шаблон для модуля:

```text
src/pages/dashboard/validator-queue/
  api/
    validatorQueueApi.ts
    validatorQueue.mapper.ts
  hooks/
    useValidatorQueueData.ts
    useValidatorQueueQuery.ts
  types/
    validator-queue.types.ts
  components/
  ValidatorQueuePage.tsx
```

Роли файлов:

- `api/*.ts` -> HTTP-запросы
- `*.mapper.ts` -> преобразование ответа сервера в формат UI
- `use...Data.ts` -> загрузка данных, ошибки, `loading`, повторный запрос
- `use...Query.ts` -> локальный поиск, сортировка, фильтрация уже загруженных данных

## Что желательно сделать перед интеграцией API

- добавить единый роутер, например `react-router`
- ввести единый HTTP-клиент
- определить типы DTO и доменные типы отдельно
- вынести повторяющиеся данные и действия из страниц
- определить стратегию работы с токеном авторизации

## Текущее состояние

Проект хорошо подходит как основа для дальнейшей продуктовой разработки, но сейчас это преимущественно UI-слой. Если начинать интеграцию бэкенда без промежуточного выделения API-слоя, страницы быстро станут перегруженными. Поэтому лучше сначала закрепить контракт модулей, описанный в документации выше.
